# 任务 2: 技术架构

## 任务概览
- 任务主题：探索多模态Agent
- 任务名称：技术架构
- 任务目标：梳理多模态Agent的典型系统架构，包括感知输入、跨模态理解、规划推理、工具调用、记忆与反馈闭环。
- 检索查询：multimodal agent architecture perception planning tool use memory

## 来源概览
- Alice Labs《AI Agent Architecture: ReAct, Tool Use & Memory Patterns》：提出生产级Agent的4层架构——感知、推理/规划、记忆、行动/工具执行；强调ReAct循环、工具接地与记忆层的重要性。
- IBM《What is AI Agent Planning?》：说明规划模块如何通过目标定义、状态表示、动作排序、优化评估支撑多步任务，并指出多Agent环境下规划复杂度更高。
- Redis《AI Agent Architecture: Build Systems That Work》：强调可靠Agent依赖架构而非提示词，架构模式选择取决于延迟、成本、可靠性约束；记忆与实时上下文基础设施是工程关键。
- Instagram资料：列出Agent七组件：感知、推理、记忆、规划、工具使用、学习、通信，可作为多模态Agent扩展组件框架。

## 任务总结
1. 多模态Agent的基础架构可抽象为“感知—理解/状态表示—规划推理—工具执行—记忆反馈”的闭环，而不是单一大模型调用。
   - Alice Labs引用Springer Nature 2025综述，将生产级Agent归纳为4个必备层：Perception/Input、Reasoning/Planning、Memory、Action/Tool Execution。
   - 对多模态场景而言，感知层需要同时接入文本、图像、音频、视频、结构化API、传感器或文档解析结果；核心价值是把异构环境信号转化为可推理的统一上下文。
   - 这意味着多模态Agent的架构重点不只是“模型能看图听音”，而是输入标准化、时间同步、模态对齐、置信度标注和上下文压缩。

2. 跨模态理解层是多模态Agent区别于普通Agent的关键中枢，需要完成模态融合、语义对齐和状态表示。
   - IBM强调规划依赖状态表示：Agent必须结构化理解当前环境、约束和上下文因素，状态表示准确性直接影响行动质量。
   - 在多模态Agent中，状态不再只是文本上下文，而可能包括视觉对象、空间关系、声音事件、用户意图、任务进度、工具返回值和历史偏好。
   - 工程上可采用多模态编码器、视觉语言模型、ASR/OCR、检索增强、知识图谱或事件流，将原始信号转为“可规划状态”。

3. 规划推理层通常采用ReAct、Plan-and-Execute或类似模式，把推理与行动交替结合，降低幻觉并支持动态修正。
   - Alice Labs指出ReAct由Yao等人在2022年提出，其核心是Thought → Action → Observation循环；相比纯链式思考，工具观测能提供外部接地，从而降低幻觉。
   - IBM将规划拆为目标定义、任务分解、动作排序、优化评估等步骤；复杂目标需要分解为可执行子任务。
   - 对多模态Agent而言，规划层需要根据不同模态证据调整路径：例如先OCR图片、再检索知识库、再调用地图或机器人控制接口，最后根据视觉反馈确认是否完成。

4. 工具调用层让多模态Agent从“理解系统”变成“行动系统”，但也引入权限、安全、成本和可靠性约束。
   - Alice Labs将Action/Tool Execution定义为外部、有副作用的执行层，典型工具包括REST API、Python REPL、浏览器自动化、SQL查询等。
   - Redis指出架构模式选择取决于延迟、成本和可靠性；这对多模态更明显，因为视觉/音频处理、外部工具和实时反馈都会放大时延与费用。
   - 工程实践应设置工具注册表、参数校验、权限边界、沙箱执行、超时重试、审计日志和人类确认节点，避免Agent在错误感知或错误规划下造成真实副作用。

5. 记忆与反馈闭环决定多模态Agent能否长期适应环境，是从“聊天机器人”升级为“智能体”的分水岭。
   - Alice Labs指出早期Agent最常见错误是跳过记忆层；没有持久记忆的系统无法跨会话保留上下文，本质上仍是单轮聊天机器人。
   - 记忆可分为短期上下文、长期语义记忆、情节记忆和程序性记忆；多模态Agent还需要存储图像/音频摘要、对象轨迹、用户交互历史和工具执行结果。
   - 反馈闭环包括工具Observation、用户反馈、环境变化、失败重试和学习更新，可支持Reflexion、ReAct迭代、缓存优化和个性化体验。

## 补充洞察
- 历史演进：Agent架构从早期规则/反应式系统，演进到LLM驱动的推理代理，再到工具增强、记忆增强、多模态感知和多Agent协作系统。
- 架构对比：Chatbot通常是无状态、单轮、无工具、无规划；Agent则是有状态、多步、多工具、可反馈修正。
- 优点：多模态Agent能处理真实世界复杂输入，具备更强上下文理解、任务执行和环境适应能力。
- 缺点：系统复杂度高，模态对齐困难，错误可能在感知—规划—工具链路中级联放大，治理与评估难度显著增加。
- 工程建议：优先设计清晰的数据流和状态模型，再选择模型；先实现最小闭环，再逐步增强记忆、工具和多模态输入。
